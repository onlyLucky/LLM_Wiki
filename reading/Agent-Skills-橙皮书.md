## 章节笔记

### 1、AI 工具的四次进化与 Skills 的诞生

每次打开 AI 工具都要把同样的事情重新解释一遍——审校标准是什么、配图风格是什么、周报格式是什么。你积累了几年的工作流程和偏好，AI 一个都记不住。每次新对话，它都是一张白纸。

这就是 Skills 要解决的问题。

AI 工具经历了四次进化：基础对话 → 上下文增强 → 工具调用 → **技能注入**。Skills 的出现让 AI 从「通用助手」变为「懂你工作的助手」。

可以类比操作系统：AI 工具是 OS，Skills 是 App。你安装一个 Skill，就像在手机上装一个 App，它让 AI 具备了新的能力。

> "每次新对话，它都是一张白纸。" · 关键句

### 2、Skill 长什么样——两层结构与三个易混淆概念

一个 Skill 本质上就是一个 SKILL.md 文件，分为两层：

**Metadata 层（永远加载，约 100 字）**：Frontmatter 里的 `name` 和 `description`。Description 决定 Skill 何时被触发——Agent 根据它判断这个 Skill 是否与当前任务相关。

**Body 层（触发时加载）**：主要指令、流程、规则。但不是越多越好——存在一个「金发姑娘区间」（Goldilocks Zone），过于详细反而降低效果。

还有第三层：**References + Scripts（按需加载）**。详细的 API 规格、完整 checklist、模板库放在独立文件里，Agent 需要时才读取。

**三个容易混淆的概念**：

与 **System Prompt** 的区别：System Prompt 是静态的全局指令，所有对话都生效。Skill 是动态加载的领域专家，只在相关时才激活。

与 **Cursor Rules** 的关系：Cursor Rules 是 Cursor 特有的实现，Skills 是跨平台的开放标准。

与 **MCP** 的区别：MCP 提供工具能力（能做什么），Skill 提供工作流程（怎么做）。两者互补，不是替代。Skills + MCP = 完整的 AI Agent 能力栈。

> "好 Skill 不是越详细越好（Goldilocks Zone）" · 关键句
> "Skills 与 MCP 的关系不是替代，而是互补" · 关键句

### 3、为什么一个 Markdown 文件能改变 AI 行为

Skills 能生效，依赖 LLM 的三个特性：

**指令遵循**：LLM 会把 SKILL.md 中的指令作为行为指南。但这里有个关键——解释「为什么」比只说「什么」更有效。

死板写法：`ALWAYS use TDD. NEVER skip tests.`
推理导向写法：「测试在 regression 到达使用者之前就拦住它。先写测试，这样你在写实现之前就知道「成功」长什么样。」

ETH Zurich 的研究发现：过度详细和限制性的指令实际上**降低**了 Agent 任务成功率。表现最好的 Agent 搭配的是精简的原则式引导。

**上下文窗口与加载**：Skill 内容被注入到上下文中，但窗口有限。太长的 Skill 会挤占留给任务本身的空间。

**条件触发**：根据 description 和用户输入匹配，决定何时加载哪个 Skill。

一个反直觉的发现：MUST/ALWAYS/NEVER 这种全大写强制指令，不如解释「为什么」有效。唯一的例外是安全约束（如「绝不把 AWS credentials commit 到版控」）。

> "ETH Zurich 研究：过度详细的指令实际上降低了 Agent 任务成功率" · 关键句

### 4、生态全景——七大平台对比与选 Skill 的标准

Skills 的开放标准平台是 agentskills.io，目前已获 20+ 主流 AI 工具支持：Claude Code（原生支持，最完整）、Cursor、GitHub Copilot、Codex、Gemini CLI、Windsurf、Cline。

选 Skill 的 5 个标准：
1. **触发准确率**：description 是否具体、是否包含触发条件
2. **指令质量**：是否解释「为什么」而不只是「什么」
3. **长度合理**：SKILL.md 是否在 500 行以内
4. **有无 Script**：确定性任务是否用 Script 而非自然语言指令
5. **跨平台兼容**：是否在多个 Agent 上都能运作

2026 年出现了一个社区现象叫「蒸馏宇宙」（The Distillation Universe）——专家把工作流程蒸馏为 Skill，其他人直接使用，形成知识共享生态。

### 5、动手装三个 Skill——四个层级与触发机制

Skill 有四个层级，决定了它的作用域：
- **全局级** `~/.skills/`：所有项目通用（如代码风格规范）
- **项目级** `.skills/`：当前项目专用（如特定框架的使用规范）
- **会话级**：当前对话临时使用
- **临时级**：一次性使用，用完即弃

安装命令示例：`npx skills add <author>/<skill-name>`

**很多人装了 Skill 却不生效**，问题往往在触发机制。你需要理解：description 决定何时触发，上下文窗口决定能加载多少内容，优先级决定多个 Skill 冲突时谁赢。

---

### 6、三种触发方式与四个常见失败原因

**三种触发方式**：
- 自动触发：根据 description 和用户输入自动匹配
- 手动触发：用户明确指定使用某个 Skill
- 混合触发：先自动推荐，用户确认后执行

**Skill 不生效的四个常见原因**：
1. Description 太模糊，Agent 无法判断何时触发
2. Skill 内容太长，被其他 context 挤掉
3. 与其他 Skill 冲突，优先级不够
4. 触发条件太窄，只对特定措辞有效

**流水线式串联才是真正的生产力倍增器**：

```
用户输入 → Skill A（调研）→ Skill B（写作）→ Skill C（审校）→ 最终产物
```

三个实战案例：用 Skill 写公众号文章（调研→大纲→初稿→审校→排版）、用 Slides Skill 做演示文稿（内容提炼→结构设计→视觉呈现）、用 Data Pro 做数据分析报告（数据清洗→分析→可视化→报告生成）。

以审校为例，拆成三遍：第一遍结构性检查（标题层级、引用格式、代码块语言标注），第二遍内容准确性（数据核实、链接有效性、术语一致性），第三遍可读性优化（长句拆分、被动语态转换、段落长度控制）。

> "Skills 像流水线一样串联——这才是真正的生产力倍增器" · 关键句

### 7、从零创建 Skill——发现隐性知识与 5 个设计原则

你日常工作中「不假思索」的决策和流程，就是最好的 Skill 材料。

问自己三个问题：
- 我每次做这件事时，第一步总是做什么？
- 我有哪些「不假思索」的判断标准？
- 新人来做这件事，我会告诉他什么？

**SKILL.md 的黄金结构**：
```yaml
---
name: my-skill
description: >
  Use this skill when [具体触发条件].
  Does [具体功能].
---
# 技能名称
## 什么时候用
## 怎么做（解释为什么）
## 规则（解释为什么）
## 参考文件
```

**5 个设计原则**：
1. Description 要具体——列出触发条件，不要写「Helps with X」
2. 解释「为什么」——让 Agent 理解意图，而非死板执行
3. 保持精简——SKILL.md < 500 行，详细内容放 reference
4. 确定性任务用 Script——文件验证、格式检查等用脚本
5. 为边界情况设计——考虑不同措辞、不同项目类型

**Description 最佳化方法**：创建 20 个评估查询（10 个应触发，10 个不应触发），每个跑 3 次，准确率低于 80% 就重写。

> "你日常工作中「不假思索」的决策和流程，就是最好的 Skill 材料" · 关键句

### 8、用 skill-creator 自动创建——四种模式与 Hill-climbing

skill-creator 有四种模式：
- **Create**：从零创建新 Skill
- **Eval**：评估现有 Skill 的质量
- **Improve**：迭代优化现有 Skill
- **Benchmark**：对比测试不同版本

核心方法论是 **Hill-climbing**：先写一个能用的版本（不要追求完美），运行测试收集反馈，根据反馈改进，重复直到质量达标。

```
初始版本 → 测试 → 分析失败 → 改进 → 再测试 → ...
```

**何时手写、何时用 skill-creator**：简单 Skill 或你很清楚流程时手写；复杂 Skill、需要迭代优化、不确定最佳写法时用 skill-creator。

> "Hill-climbing：先写能用的版本，再逐步优化——不要追求一次完美" · 关键句

### 9、六种高级设计模式——从检查清单到思维蒸馏

**模式 1：检查清单型（Checklist）**
确保不遗漏关键步骤。适用：代码审查、部署检查、安全审计。

**模式 2：多方案选择型（Options）**
提供 2-3 个方案各有优劣，让 Agent 根据 context 选择。适用：技术选型、设计方案。

**模式 3：多阶段流水线型（Pipeline）**
把复杂任务拆成阶段，每个阶段一个 Skill，串联执行。关键：每个 Skill 的输出格式要标准化。适用：内容创作、数据分析、软件开发。

**模式 4：外部 API 集成型（Integration）**
用 Script 封装 API 调用，Skill 提供使用指南。适用：数据查询、第三方工具集成。

**模式 5：多 Agent 协作型（Swarm）**
每个 Agent 一个角色，通过 Skill 定义协作规则。适用：大型项目、需要多人协作的任务。

**模式 6：思维蒸馏型（Distillation）**
把专家经验蒸馏为可复用的推理过程。方法：观察专家工作→记录决策和理由→转化为条件判断和行动指令→测试→迭代。适用：领域专家的工作流程、决策过程。

| 模式 | 适用场景 | 复杂度 |
|------|----------|--------|
| Checklist | 确保不遗漏 | 低 |
| Options | 多方案选择 | 低 |
| Pipeline | 复杂任务分阶段 | 中 |
| Integration | 连接外部服务 | 中 |
| Swarm | 多 Agent 协作 | 高 |
| Distillation | 专家经验固化 | 高 |

> "Distillation Pattern：把专家经验蒸馏为可复用的推理过程" · 关键句

### 10、Skills 的未来——从个人工具到经济生态

**从个人到团队**：Skills 可以在团队内共享，形成团队的「知识库」。

**Skills 经济**：创作者生态，经验可以被定价和交易。三个层次：个人层（提高效率）→ 团队层（共享知识）→ 生态层（社区交易）。

**Agent 自己创建 Skills**：未来的 Agent 可以观察你的工作过程，自动提取关键决策和流程，生成新的 Skill，持续优化。

> "Skills 不只是一个技术特性，它可能是 AI 时代「把工作经验变成可复用数字资产」的起点。" · 关键句

### 附录：模板、索引与资源

**SKILL.md 模板（可直接复制使用）**：
```yaml
---
name: my-skill-name
description: >
  Use this skill when [触发条件].
  Does [具体功能].
  Outputs [输出格式].
---
# 技能名称
## 什么时候用
## 怎么做
## 规则
## 参考文件
```

**发布前 Checklist**：
- Description 具体、包含触发条件
- Body 解释规则背后的 WHY，不只 WHAT
- SKILL.md < 500 行；详细参考文件独立存放
- 确定性任务用 Script 而非自然语言指令
- 用 2-3 个真实提示测试过，并与无技能基线对比
- 触发准确率测试过（≥10 个应触发 & ≥10 个不应触发的查询）
- 无硬编码路径、API key 或环境特定值
- 能在多个 Agent 上运作

## 最大的 3 个收获

1. **好 Skill 不是越详细越好**：ETH Zurich 研究证实，过度详细的指令降低 Agent 任务成功率。存在「金发姑娘区间」，精简的原则式引导效果最好。
2. **Skills 是可复用的数字资产**：知识工作者的价值从「执行」转向「设计流程」——把自己的工作经验蒸馏为 SKILL.md 文件。
3. **六种设计模式**：从简单的 Checklist 到复杂的 Distillation，覆盖了创建高质量 Skill 的完整方法论。
