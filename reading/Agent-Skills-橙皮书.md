## 为什么读这本书

这本书解决的核心痛点是 **AI 的失忆症**：你积累了几年的审校标准、配图风格、周报格式，AI 一个都记不住。每次新对话它都是一张白纸。Skills 的出现让这个问题有了优雅的解法——把你的工作流程写成一份 `SKILL.md` 文件，AI 就能按你的方式工作。不需要写代码，用自然语言就行。

全书四个部分对应四个能力阶段：**概念认知（1-3章）→ 上手实战（4-6章）→ 创作开发（7-9章）→ 未来展望（10章）**。适合已经在用 Claude Code、Cursor 或其他 AI 编程工具、想从「对话式使用」升级到「系统化协作」的读者。

## 章节笔记

### 1、AI 工具的四次进化与 Skills 的诞生

#### AI 能力的四次进化路径

AI 工具的能力演进可以划分为四个明确阶段，每一步都在解决上一阶段遗留的核心瓶颈。
理解这个进化路径能帮助你判断当前工具处于哪个阶段，以及下一步该投资什么能力。很多人还在用第一阶段的方式使用已经进化到第四阶段的工具。

| 阶段 | 名称 | 核心能力 | 典型产品 | 用户体验 |
|------|------|----------|----------|----------|
| 第一次 | 基础对话 | 单轮问答 | 早期 ChatGPT | 问一句答一句，无法处理复杂任务 |
| 第二次 | 上下文增强 | 长上下文 + 记忆 | ChatGPT 持久记忆、Claude Projects | 能记住你说过什么，但不知道你想要什么格式 |
| 第三次 | 工具调用 | Function Calling / MCP | Claude Code、Cursor | 能操作文件、API、数据库，但不知道你的工作流程 |
| 第四次 | 技能注入 | Skills / SKILL.md | Claude Code + Skills、Codex CLI | 既能操作工具，又知道用你的标准和流程来操作 |

在第三阶段，你可以让 AI 帮你写一篇文章，它能搜索资料、写草稿。但到了第四阶段，你装一个「公众号审校 Skill」，AI 会自动按你的三遍审校流程（结构→内容→可读性）来打磨文章，输出风格一致。
很多人把第三阶段的「工具调用」和第四阶段的「技能注入」混为一谈。关键区别在于——工具调用给了 AI 一把锤子，技能注入告诉它用锤子的正确姿势和建造流程。

#### 操作系统类比：AI = OS，Skills = App

作者提出了一个直观的心智模型——把 AI 工具类比为操作系统，Skills 类比为 App。
这个类比帮你快速理解 Skills 的生态定位。就像操作系统本身功能有限，装了 App 才能做具体事情一样，AI 模型本身是通用的，装了 Skills 才能在特定场景下表现出专家级水平。

| 操作系统概念 | AI/Skills 对应 |
|-------------|---------------|
| 操作系统（macOS、Windows） | AI 平台（Claude Code、Cursor） |
| App Store | agentskills.io、社区 Skills 仓库 |
| App（应用程序） | Skill（技能文件） |
| App 安装 | 把 SKILL.md 放到 .skills/ 目录 |
| App 卸载 | 删除 Skill 文件夹 |
| App 更新 | 更新 SKILL.md 内容 |
| 系统级 App | 全局级 Skill（~/.skills/） |
| 项目级配置 | 项目级 Skill（.skills/） |

你的 Mac 上有 macOS，但它不会自动帮你做 PPT。你装了 Keynote（App），它就会了。同样，Claude Code 是 AI 平台，你装了 slides.skill，它就知道怎么按你的风格做演示文稿。
这个类比有一个关键不同——App 是编译过的二进制代码，而 Skill 是纯自然语言的 Markdown 文件。这意味着任何人都能创建 Skill，门槛远低于开发 App。这是 Skills 生态能快速繁荣的根本原因。

#### Skills 的串联工作方式

多个 Skills 可以像流水线一样串联工作，前一个 Skill 的输出直接成为下一个 Skill 的输入。
单个 Skill 解决的是单点问题，串联 Skills 解决的是完整工作流问题。这才是 Skills 真正的威力——把你的整个工作流程自动化。
使用文件系统作为"传送带"。Skill A 产出一个文件（如 `draft.md`），Skill B 读取这个文件继续处理，Skill C 再接着处理。文件就是 Skill 之间的"接力棒"。

```
用户输入："写一篇关于 AI Agent 的公众号文章"
→ Skill 1: topic-research（选题调研）
  输出: research-notes.md
→ Skill 2: outline-generator（大纲生成）
  输入: research-notes.md
  输出: article-outline.md
→ Skill 3: draft-writer（初稿写作）
  输入: article-outline.md + research-notes.md
  输出: draft-v1.md
→ Skill 4: proofreader（三遍审校）
  输入: draft-v1.md
  输出: draft-v2.md
→ Skill 5: image-generator（配图生成）
  输入: draft-v2.md
  输出: images/ 目录
→ 最终产物: 一篇配好图的、经过三遍审校的公众号文章
```

### 2、Skills 的本质：不是提示词，是工作流

#### 拆解一个最简单的 Skill

一个最简单的 Skill 就是一个文件夹，里面有一个 `SKILL.md` 文件。这个文件包含 YAML frontmatter（元数据）和 Markdown 正文（指令）。

```markdown
---
name: code-review
description: 当用户请求代码审查时使用。遵循公司的代码规范进行审查。
---

# 代码审查规范

## 审查流程
1. 先检查代码风格是否符合规范
2. 再检查逻辑正确性
3. 最后检查性能问题

## 输出格式
每个问题标注严重等级：🔴 严重 / 🟡 建议 / 🟢 可选
```

理解最小结构是创建和修改 Skill 的基础。很多人以为需要复杂的代码，其实 frontmatter 就两个必需字段：`name` 和 `description`。

#### Skills 的两层渐进式披露结构

Skills 采用了「渐进式披露」（Progressive Disclosure）的三层加载架构，这是它区别于 System Prompt 的核心设计。

| 层级 | 内容 | 加载时机 | Token 消耗 | 说明 |
|------|------|----------|-----------|------|
| 第一层：Metadata | name + description（frontmatter） | Agent 启动时，始终加载 | ~100 token/个 | 即使装了 50 个 Skills，初始消耗也只有 ~5,000 token |
| 第二层：Body | SKILL.md 正文（指令、规则、流程） | Agent 判断任务相关时才加载 | 1,000~5,000 token | 按需加载，不占用初始上下文 |
| 第三层：References/Scripts | 引用文件、脚本、模板 | Body 中明确引用时才加载 | 视文件大小而定 | 最大化节省上下文空间 |

这个设计解决了一个关键矛盾——你想给 AI 足够多的指令（Skill 多），但上下文窗口有限。渐进式披露让 50 个 Skills 的初始成本只有 5,000 token，而不是全部加载的 250,000 token。
你装了 20 个 Skills。AI 启动时只读到 20 条 description（每条约 100 token = 2,000 token）。当你说"帮我审校这篇文章"，AI 匹配到 `proofreader` Skill，才加载它的完整指令（2,000 token）。其他 19 个 Skill 的 Body 永远不加载。
把所有指令都写在 frontmatter 的 description 里。description 应该简短（一句话说明"什么时候用"），详细指令放在 Body 里。

#### 与 System Prompt 的根本区别

Skills 和 System Prompt 都是给 AI 的指令，但它们的设计哲学和使用方式完全不同。

| 维度 | System Prompt | Skill |
|------|--------------|-------|
| 作用范围 | 全局、始终生效 | 按需触发、领域专属 |
| 加载方式 | 启动时全量加载 | 渐进式披露 |
| 内容定位 | 通用行为规范 | 特定任务的专业流程 |
| 修改方式 | 需要改平台配置 | 添加/删除文件夹即可 |
| 可移植性 | 绑定特定 AI 产品 | 跨平台通用（开放标准） |
| 触发条件 | 无条件始终生效 | 由 description 匹配触发 |
| 组织方式 | 一个巨大的文本块 | 多个独立的 Skill 文件 |
| 调试方式 | 改了全看效果 | 单个 Skill 独立测试 |
| 社区共享 | 不便分享 | GitHub 一键共享 |
| 协作方式 | 每人自己配 | 团队共享 .skills/ 目录 |

System Prompt 像公司全员邮件，所有人都收到；Skill 像按需查阅的 SOP 手册，只有需要时才打开。
System Prompt 说"你是一个有帮助的助手"（全局行为）。Skill 说"当用户请求代码审查时，按以下 5 个步骤检查"（特定任务流程）。两者互补——System Prompt 定义"你是谁"，Skill 定义"具体怎么做"。

#### 与 MCP 的关系：互补而非替代

MCP（Model Context Protocol）和 Skills 是 AI 能力栈的两个不同层次，一个提供"手"（工具能力），一个提供"脑"（工作流程）。

| 维度 | MCP | Skills |
|------|-----|--------|
| 核心功能 | 提供工具能力（能做什么） | 提供工作流程（怎么做） |
| 本质 | 协议/接口 | 指令/流程 |
| 比喻 | 锤子、螺丝刀 | 说明书、SOP |
| 加载方式 | 急切加载（Eager Loading） | 惰性加载（Lazy Loading） |
| 技术门槛 | 需要写代码 | 纯自然语言 |
| 示例 | 连接数据库、读写文件、调用 API | 审校流程、代码规范、数据分析模板 |
| Token 成本 | 每次对话都加载工具定义 | 按需加载指令 |

很多人把 MCP 和 Skills 搞混，以为有了 MCP 就不需要 Skills，或者反过来。实际上它们是互补的——MCP 给你能力，Skills 告诉你怎么用能力。

- **MCP 提供的能力**：`database-query` 工具可以连接 MySQL、执行 SQL、返回结果
- **Skill 提供的流程**：`employee-analysis` Skill 告诉 AI "先查员工表结构，再按部门分组统计，输出格式用 Markdown 表格，保留两位小数"
- **两者协作**：AI 按 Skill 的流程，调用 MCP 的工具，完成任务

#### 与 Cursor Rules 的关系

Cursor Rules 是 Cursor 编辑器的特有实现，Skills 是跨平台的开放标准。两者有历史渊源但走向不同。

| 维度 | Cursor Rules | Skills (SKILL.md) |
|------|-------------|-------------------|
| 平台 | Cursor 专属 | 跨 20+ 平台通用 |
| 文件格式 | .cursorrules 文件 | SKILL.md 文件夹 |
| 触发方式 | 始终生效或项目级 | description 匹配触发 |
| 标准化程度 | 专有格式 | 开放标准 (agentskills.io) |
| 渐进式披露 | 无 | 有（三层架构） |
| 社区生态 | 小 | 快速增长中 |

如果你已经在用 Cursor Rules，迁移到 Skills 不仅能获得跨平台兼容性，还能获得渐进式披露带来的 token 效率提升。

#### 真实案例：三遍审校 Skill

花叔创建的三遍审校 Skill，是书中最核心的实战案例，展示了如何把人类专家的审校流程蒸馏成 Skill。

| 遍次 | 检查重点 | 具体动作 | 输出 |
|------|----------|----------|------|
| 第一遍 | 结构性检查 | 检查标题层级、段落逻辑、开头结尾、论点论据结构 | 结构修改建议 |
| 第二遍 | 内容准确性 | 核实数据来源、检查引用准确性、逻辑自洽性 | 内容修正清单 |
| 第三遍 | 可读性优化 | 调整句式、消除冗余、优化过渡、统一风格 | 最终润色版 |

它完美展示了 Skills 的核心价值——把你"不假思索"的审校习惯（隐性知识）变成 AI 可以精确执行的流程（显性知识）。每遍的检查项不是笼统的"检查质量"，而是具体到"标题层级是否一致""论点是否有数据支撑"这种可执行的指令。

1. 为什么分三遍而不是一遍？——因为混合检查会导致遗漏，人类专家也是分层检查的
2. 每遍为什么有明确的检查清单？——因为"好好检查"是无效指令，必须具体化
3. 为什么要在 Body 里写"为什么这样检查"？——因为解释原因比下命令更有效（金发姑娘原则）

#### 为什么 Skills 能改变 AI 行为

Skills 通过指令遵循（Instruction Following）机制改变 AI 的行为方式。AI 模型天生会遵循上下文中明确出现的指令，Skill 的 Body 被加载后，就成为 AI 需要遵循的指令集。
当 Skill 的 Body 被注入到上下文窗口中，它和用户消息一起构成 AI 的"思考背景"。AI 在生成回答时，会同时考虑用户需求和 Skill 中的指令。这不是魔法，而是大语言模型的基本工作原理——上下文越明确，输出越可控。

### 3、Skills 背后的原理：指令遵循与上下文工程

#### 指令遵循的底层机制

指令遵循是大语言模型的基本能力——模型会优先遵循上下文中明确、具体的指令。Skills 正是利用这个机制工作的。
理解指令遵循，你才能写出有效的 Skill。模糊的指令产生模糊的结果，具体的指令产生具体的输出。

| 因素 | 影响程度 | 说明 |
|------|----------|------|
| 指令的明确性 | ★★★★★ | "用表格输出" > "清楚地呈现" |
| 指令的位置 | ★★★★☆ | 开头和结尾的指令权重高于中间 |
| 指令的重复度 | ★★★☆☆ | 适度重复强化，但过度重复会被忽略 |
| 指令的理由 | ★★★★★ | 解释"为什么"比下命令更有效 |
| 指令的冲突 | ★★☆☆☆ | 冲突指令会让模型困惑，需要优先级机制 |

MUST/ALWAYS/NEVER 这类强制命令的效果不如解释"为什么"。告诉 AI "不要这样做，因为会导致 XX 问题"比"永远不要这样做"更有效。唯一的例外是安全约束——安全相关的指令用强制语气是合适的。

#### 上下文窗口与 Skill 的加载策略

上下文窗口是 AI 能同时"看到"的信息量上限。Skills 的三层架构就是围绕上下文窗口的限制设计的。

```
1. Agent 启动
   └→ 扫描 .skills/ 目录下所有 Skill 文件夹
   └→ 读取每个 SKILL.md 的 frontmatter（name + description）
   └→ 将所有 description 注入系统提示词（~100 token/个）

2. 用户发送消息
   └→ Agent 将用户消息与所有 description 进行匹配
   └→ 如果匹配到某个 Skill：
       └→ 读取该 Skill 的 SKILL.md 完整内容到上下文
       └→ 如果 Body 中引用了 references/ 或 scripts/：
           └→ 按需读取引用文件

3. Agent 执行任务
   └→ 按照 Skill Body 中的指令执行
   └→ 可能调用 MCP 工具、读写文件等

4. 输出结果
   └→ 按 Skill 中定义的格式输出
```

- 每个 Skill 的 metadata 层消耗约 100 token
- 50 个 Skills 的初始上下文消耗约 5,000 token
- 单个 Skill 的 Body 通常在 1,000~5,000 token
- 对比：一个 MCP server 的工具定义可能消耗 20,000+ token

上下文窗口是 AI 的"工作记忆"，装了太多指令就挤掉了任务空间。Skills 的渐进式披露让 50 个 Skills 只占 5,000 token 初始空间，而 System Prompt 方式需要全部加载。

#### 条件触发机制

Skills 不是无条件加载的，而是由 AI 根据任务内容和 Skill 的 description 进行语义匹配后决定是否触发。

1. AI 接收到用户消息
2. AI 扫描系统提示词中所有 Skill 的 description
3. AI 判断用户任务与哪个（哪些）Skill 的 description 最匹配
4. 匹配到的 Skill 的 Body 被加载到上下文
5. AI 按照 Body 中的指令执行

这个匹配是基于 AI 的自然语言理解，不是关键词匹配。所以 description 的质量直接影响触发准确率。

1. 准备 20 个测试查询：10 个应该触发该 Skill + 10 个不应该触发
2. 每个查询跑 3 次
3. 如果准确率低于 80%，重写 description
4. description 应该描述"什么时候用"，而不是"这个 Skill 是什么"

#### 执行模型

Skill 的执行遵循一个清晰的模型：**发现 → 匹配 → 加载 → 执行 → 输出**。

| 阶段 | 行为 | 关键设计 |
|------|------|----------|
| 发现 | 扫描 .skills/ 目录 | 目录结构决定可见性 |
| 匹配 | description 语义匹配 | description 质量决定准确率 |
| 加载 | 读取 Body + 引用文件 | 渐进式披露控制 token 成本 |
| 执行 | 按指令生成输出 | Body 中的指令决定输出质量 |

#### 金发姑娘区间：好 Skill 不是越详细越好

ETH Zurich 的研究发现，过度详细的指令反而会降低 AI Agent 的任务成功率。存在一个"恰到好处"的指令详细度区间——就像金发姑娘的粥，不能太热也不能太冷。

| 指令详细程度 | 任务成功率 | 原因 |
|-------------|-----------|------|
| 极简（一句话） | 低 | 信息不足，AI 自由发挥 |
| 精简+原则引导 | **最高** | 给出方向和边界，AI 有发挥空间 |
| 详细逐步指令 | 中等偏高 | 机械执行，但可能遗漏例外情况 |
| 过度详细（每步都指定） | **降低** | AI 过度聚焦指令本身，忽略任务目标 |
| 超长指令集 | 最低 | 上下文污染，AI 处理不过来 |

过度详细的指令导致两个问题——
1. **指令过拟合**：Agent 把每条指令都当作必须执行的任务，而不是参考建议，导致在不适用的指令上浪费时间
2. **上下文挤占**：过多指令挤掉了实际任务需要的上下文空间

- ✅ 好的做法："审校文章时关注结构、内容准确性、可读性三个层面。结构层面检查标题层级和段落逻辑……"
- ❌ 不好的做法："第一步，打开文件。第二步，读取第一段。第三步，检查第一段的第一个句子的主语是否……"（过度微观管理）

#### Token 经济学：Skill 的成本管理

每个 Skill 都有 token 成本。Skill 越长，消耗的 token 越多，留给实际任务的上下文空间就越少。这就是 Token 经济学。

```
总上下文窗口（假设 200K token）
├── 系统提示词（平台级）        ~10,000 token
├── 所有 Skill 的 description   ~5,000 token（50 个 Skill）
├── 激活 Skill 的 Body          ~3,000 token（1~2 个 Skill）
├── MCP 工具定义               ~20,000 token
├── 对话历史                    ~30,000 token
├── 任务内容                    ~50,000 token
└── 剩余空间（推理+输出）       ~82,000 token
```

1. **保持 Body 精简**：SKILL.md 正文控制在 500 行、5,000 token 以内
2. **用 references/ 分流**：不常用的详细信息放在引用文件中，按需加载
3. **用 scripts/ 代替文本指令**：确定性任务用脚本执行，比文本指令更节省 token
4. **description 要短**：每个 description 控制在 1~2 句话

#### Skill 冲突处理

当你安装了多个 Skills，有时它们的指令会冲突。比如两个 Skill 都要求"用 Markdown 表格输出"，但格式不同。

| 冲突类型 | 例子 | 处理策略 |
|----------|------|----------|
| 格式冲突 | 两个 Skill 定义不同的输出格式 | 在 Skill 中明确优先级或合并格式 |
| 流程冲突 | 一个要求先写大纲再写正文，另一个要求直接写 | 在 description 中明确适用场景 |
| 原则冲突 | 一个要求简洁，另一个要求详细 | 在 Skill 中注明"当与 XX Skill 共用时"的特殊规则 |

尽量让每个 Skill 职责单一。如果有冲突，说明你可能需要把它们拆分成不重叠的 Skill，或者在 description 中明确各自的适用场景。

### 6、用好 Skills 的实战技巧

#### 三种触发方式

| 触发方式 | 说明 | 适用场景 |
|----------|------|----------|
| 自动触发 | AI 根据 description 自动匹配 | 日常重复任务 |
| 手动触发 | 用户明确要求使用某个 Skill | 你知道要用哪个 Skill |
| 混合触发 | 自动触发 + 手动微调 | 复杂多步骤任务 |

#### 四个常见失败原因及解决方案

**失败原因 1：description 模糊**

| 问题描述 | 错误示例 | 正确示例 |
|----------|----------|----------|
| description 无法区分触发场景 | "帮助写作" | "当用户请求公众号文章审校时使用，关注结构、内容准确性、可读性三个层面" |

准备 20 个查询（10 应触发 + 10 不应触发），每个跑 3 次。准确率 < 80% → 重写 description。
**失败原因 2：Skill 太长被挤掉**

- 症状：AI 有时遵守 Skill，有时不遵守
- 原因：Skill 的 Body 太长，加上任务内容后超过上下文窗口，AI 自动"遗忘"部分内容
- 解决：控制 Body 在 500 行以内；用 references/ 分流详细内容

**失败原因 3：与其他 Skill 冲突**

- 症状：AI 行为不一致，有时用 A Skill 的规则，有时用 B Skill 的规则
- 原因：两个 Skill 的 description 重叠，或者 Body 中有矛盾指令
- 解决：明确每个 Skill 的 description 边界，避免重叠

**失败原因 4：触发条件太窄**

- 症状：你认为应该触发的场景，AI 没有触发
- 原因：description 写得太具体，稍微变化的表述就无法匹配
- 解决：description 用更通用的表述，覆盖更多触发场景

#### 流水线串联的实现方式

文件系统是 Skill 之间的"传送带"。前一个 Skill 产出文件，后一个 Skill 读取文件继续处理。

1. Skill A 完成任务，输出文件到指定路径（如 `output/draft.md`）
2. Skill B 的 Body 中指明"读取 `output/draft.md` 作为输入"
3. Skill B 完成任务，输出文件到下一个路径（如 `output/reviewed.md`）
4. 依此类推

```
选题调研 Skill → outline.md
大纲生成 Skill → article-outline.md
初稿写作 Skill → draft-v1.md
三遍审校 Skill → draft-v2.md（结构修改）→ draft-v3.md（内容修正）→ draft-final.md（润色）
配图生成 Skill → images/
最终产物: 一篇完整的公众号文章
```

#### 审校环节拆解

以三遍审校 Skill 为例，展示如何把一个完整的审校流程拆解为可执行的 Skill 指令。

- [ ] 标题层级是否一致（h1→h2→h3，不跳级）
- [ ] 每个段落是否有明确的主题句
- [ ] 开头是否能在 3 秒内抓住读者
- [ ] 结尾是否有行动号召或总结升华
- [ ] 论点之间的逻辑关系是否清晰（并列/递进/因果）

- [ ] 所有数据和引用是否有来源
- [ ] 逻辑是否自洽（前文说 A，后文不能说 ¬A）
- [ ] 是否有过度概括的表述（"所有人都""从来没有"）
- [ ] 专业术语是否准确

- [ ] 句式是否多样（避免全用长句或全用短句）
- [ ] 是否有冗余表述可以删除
- [ ] 段落之间的过渡是否自然
- [ ] 全文风格是否统一

#### Slides Skill 案例

用 Skills 做演示文稿的完整流程，展示了如何把"做 PPT"这个复杂任务分解为可重复的步骤。

1. 根据主题确定 5~7 个核心要点
2. 每个要点对应 1~2 页幻灯片
3. 每页遵循"标题 + 3 个要点 + 配图/数据"结构
4. 第一页是封面（标题 + 副标题 + 作者）
5. 最后一页是总结和行动号召

没有 Skill 时，AI 做的 PPT 格式随机、内容结构不一致。有了 Skill，每次产出的 PPT 都遵循你的风格标准。

#### Data Pro 数据分析案例

用 Skills 做数据分析报告的案例，展示了如何让 AI 按照专业的数据分析流程工作。

1. 数据概览：行数、列数、数据类型、缺失值统计
2. 数据清洗：处理缺失值、异常值、格式统一
3. 探索性分析：分布、相关性、趋势
4. 深度分析：根据业务问题选择分析方法
5. 可视化：选择合适的图表类型
6. 结论输出：用 Markdown 表格 + 图表呈现

| 排查项 | 检查方法 | 常见问题 |
|--------|----------|----------|
| 文件位置 | 检查 .skills/ 目录路径 | 放错了目录层级 |
| 文件名 | 确认是 SKILL.md（大小写） | 写成了 skill.md 或 SKILLS.md |
| frontmatter | 确认有 name 和 description | YAML 格式错误 |

### 7、创建你的第一个 Skill：从隐性知识到显性流程

#### 发现隐性知识：问自己三个问题

创建 Skill 的第一步不是写 SKILL.md，而是发现自己"不假思索"的工作习惯——这就是隐性知识。

| 问题 | 目的 | 示例答案 |
|------|------|----------|
| 1. 我每次做这类任务时，第一步做什么？ | 发现流程起点 | "先看需求文档，再看现有代码结构" |
| 2. 我做决策时，脑子里在想什么？ | 发现决策逻辑 | "如果改动超过 20 行，我会先写设计文档" |
| 3. 我检查质量时，重点看什么？ | 发现质量标准 | "先看有没有硬编码，再看错误处理" |

你之所以能高效工作，是因为你积累了大量"不假思索"的决策模式。Skill 的价值就是把这些模式变成 AI 可以执行的显性指令。

#### SKILL.md 完整结构详解

```markdown
---
name: my-expert-skill
description: 当用户请求 XX 时使用此 Skill。专注于 YY 场景，产出 ZZ 格式。
---

# Skill 名称

## 什么时候用
<!-- 明确适用场景和不适用场景 -->

## 工作流程
<!-- 分步骤的具体指令 -->
### 步骤 1：XX
<!-- 具体做什么、怎么做、为什么这样做 -->

### 步骤 2：YY
...

## 规则与约束
<!-- 必须遵守的规则，解释 why -->

## 输出格式
<!-- 最终产出的格式要求 -->

## 参考文件
<!-- 引用 references/ 中的文件 -->
- 参考 `references/style-guide.md` 了解排版规范
- 参考 `references/examples.md` 查看输出示例
```

| 部分 | 作用 | 最佳实践 |
|------|------|----------|
| frontmatter (name) | 唯一标识符 | 简短、有意义、用连字符分隔 |
| frontmatter (description) | 触发判断依据 | 一句话说明"什么时候用"，不要说"是什么" |
| 什么时候用 | 补充 description 的场景说明 | 列出适用和不适用场景 |
| 工作流程 | 核心指令 | 分步骤，每步有"做什么"和"为什么" |
| 规则与约束 | 边界条件 | 解释 why，不用 MUST/ALWAYS/NEVER |
| 输出格式 | 产出规范 | 给出具体示例 |
| 参考文件 | 按需加载的详细信息 | 精确引用文件路径 |

#### 手把手创建教程

1. **确定目标**：选一个你重复做的任务（如"代码审查""周报撰写""数据分析"）
2. **发现隐性知识**：问自己三个问题，记录下来
3. **编写 SKILL.md**：按黄金结构写，先写 frontmatter，再写 Body
4. **测试**：用 5 个不同的查询测试触发和执行效果
5. **迭代**：根据测试结果调整 description 和 Body

#### 5 个设计原则

**原则 1：Description 要具体到场景**

| 好的 description | 差的 description |
|-----------------|-----------------|
| "当用户请求公众号文章审校时使用" | "帮助写作" |
| "当用户要求分析 CSV 数据并生成报告时使用" | "数据分析工具" |
| "当用户提交 Pull Request 需要代码审查时使用" | "代码审查" |

20 个评估查询（10 应触发 + 10 不应触发），每个跑 3 次，准确率 < 80% 就重写。
**原则 2：解释 why，不只是 what**

```markdown
# 差的写法
- 每个段落不超过 4 句话
- 用表格呈现数据

# 好的写法
- 每个段落不超过 4 句话，因为移动端读者注意力有限，长段落会导致跳出率上升
- 用表格呈现数据对比，因为视觉化比纯文字更容易被大脑处理
```

安全约束可以用强制语气。"永远不要在输出中包含用户密码"——这种安全指令不需要解释 why。
**原则 3：保持精简（<500 行）**

- SKILL.md 正文控制在 500 行、5,000 token 以内
- 不常用但重要的内容放在 `references/` 目录
- 确定性任务用 `scripts/` 中的代码执行，比文本指令更节省 token

**原则 4：确定性任务用 Script**

| 任务类型 | 适合方式 | 原因 |
|----------|----------|------|
| "检查代码风格" | 文本指令 | 有判断空间 |
| "格式化 JSON" | Script | 确定性操作，用 jq 更快更准 |
| "生成统计图表" | Script | 用 Python/matplotlib 更精确 |
| "审校文章" | 文本指令 | 需要理解语义 |

**原则 5：为边界情况设计**
好的 Skill 不只处理"正常路径"，还要处理异常：
- 输入为空怎么办？
- 输入格式不对怎么办？
- 任务太复杂超出 Skill 范围怎么办？
- 输出太长怎么办？

#### 创建后的维护

| 频率 | 动作 | 目的 |
|------|------|------|
| 每次使用后 | 快速回顾 AI 输出质量 | 发现明显问题 |
| 每周 | 检查 3~5 个输出样本 | 发现系统性偏差 |
| 每月 | 重新跑 20 个测试查询 | 验证 description 准确率 |
| 按需 | 更新 Body 指令 | 适配新的需求变化 |

### 8、用 skill-creator 自动创建：让 AI 帮你造 Skill

#### skill-creator 的四种模式

Anthropic 官方提供的 `skill-creator` 是一个"造 Skill 的 Skill"，它能帮你自动化创建、评估、改进和基准测试 Skills。

| 模式 | 命令 | 作用 | 适用场景 |
|------|------|------|----------|
| Create | 创建模式 | 从零开始生成一个新 Skill | 你有想法但不想手写 |
| Eval | 评估模式 | 评估现有 Skill 的质量 | 你写了一个 Skill，想检查质量 |
| Improve | 改进模式 | 分析问题并优化现有 Skill | Skill 效果不理想 |
| Benchmark | 基准测试 | 用标准化测试集评估 Skill | 想要客观的质量指标 |

1. 你描述你的工作流程（自然语言）
2. skill-creator 分析你的描述，提取关键决策点
3. 自动生成 SKILL.md 的 frontmatter 和 Body
4. 生成 description 的初始版本
5. 给出优化建议

1. 读取你的 SKILL.md
2. 分析 description 的清晰度和触发准确度
3. 分析 Body 的指令质量
4. 检查是否符合 5 个设计原则
5. 给出评分和改进建议

1. 读取你的 SKILL.md 和 Eval 结果
2. 针对发现的问题逐一修改
3. 重写模糊的 description
4. 精简过长的 Body
5. 增加缺失的边界情况处理
6. 输出改进后的 SKILL.md

1. 使用标准化测试场景
2. 对比改进前后的 Skill 表现
3. 输出量化指标（触发率、执行准确率、输出质量评分）

#### Hill-climbing 优化策略

Hill-climbing（爬山法）是 skill-creator 使用的迭代优化策略——每次只改一个变量，测试效果，保留改进，循环往复。

```
初始 Skill（v1）
│
├── 评估 → 发现 3 个问题
│
├── 修改问题 1 → 测试 → 如果更好，保留
│
├── 修改问题 2 → 测试 → 如果更好，保留
│
├── 修改问题 3 → 测试 → 如果更好，保留
│
└── 输出优化版 Skill（v2）
    │
    └── 继续评估 → 循环...
```

每次只改一个变量。如果同时改了 3 个地方，效果变好了你不知道是哪个改进起了作用，效果变差了也不知道是哪个改坏了。

#### 手写 vs 自动创建的决策

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 你很清楚流程，想快速创建 | 手写 | 你已有完整的心智模型 |
| 你有流程但不确定如何结构化 | skill-creator | 它帮你结构化 |
| 你有代码经验，需要脚本 | 手写 + skill-creator 辅助 | 手写 scripts，自动优化 description |
| 你是非技术用户 | skill-creator | 零代码门槛 |
| 你想优化现有 Skill | skill-creator Improve 模式 | 自动分析和改进 |

#### 思维框架：从"做什么"到"怎么想"

创建 Skill 不只是记录步骤，更是把你的思维方式结构化。skill-creator 引导你从三个层次思考：

| 层次 | 问题 | 输出 |
|------|------|------|
| 表层：做什么 | 具体步骤是什么？ | 工作流程 |
| 中层：怎么决策 | 遇到选择时怎么判断？ | 条件判断规则 |
| 底层：为什么 | 为什么这样做而不是那样？ | 设计原则和原因 |

### 9、高级设计模式：从单点到系统

#### 检查清单型（Checklist Pattern）

最简单、最常用的模式。把质量标准转化为逐项检查的清单。
代码审查、文章审校、PR 审核、安全检查

```markdown
# 代码审查检查清单

## 安全性检查
- [ ] 是否有硬编码的密钥或密码
- [ ] SQL 查询是否使用参数化
- [ ] 用户输入是否做了校验和清洗

## 代码质量检查
- [ ] 函数是否遵循单一职责原则
- [ ] 是否有未使用的变量或导入
- [ ] 错误处理是否完整

## 性能检查
- [ ] 是否有 N+1 查询问题
- [ ] 大数据量操作是否有分页
- [ ] 是否有不必要的循环嵌套
```

每项检查都要说明"为什么检查这个"以及"发现问题后怎么修"。

#### 多方案选择型（Options Pattern）

当任务有多种可能的处理方式时，让 AI 评估每种方案的优劣，给出推荐。
技术方案选型、架构设计、产品决策

```markdown
# 技术方案评估

## 流程
1. 理解需求的核心约束（性能/成本/时间/团队能力）
2. 提出 2~3 个可行方案
3. 每个方案分析：
   - 优点（至少 2 个）
   - 缺点（至少 2 个）
   - 适用场景
   - 风险评估
4. 给出推荐方案，说明推荐理由
5. 给出备选方案的切换条件
```

#### 多阶段流水线型（Pipeline Pattern）

把复杂任务分解为多个阶段，每个阶段有明确的输入、处理和输出。
长文写作、数据分析报告、项目规划

```markdown
# 公众号文章写作流水线

## 阶段 1：选题调研（输入：主题关键词）
输出：research-notes.md（含竞品文章分析、读者痛点、差异化角度）

## 阶段 2：大纲生成（输入：research-notes.md）
输出：outline.md（含标题、各节标题、每节核心论点）

## 阶段 3：初稿写作（输入：outline.md + research-notes.md）
输出：draft-v1.md

## 阶段 4：三遍审校（输入：draft-v1.md）
输出：draft-final.md

## 阶段 5：配图（输入：draft-final.md）
输出：images/ 目录 + 最终文章
```

#### 外部 API 集成型（Integration Pattern）

当 Skill 需要调用外部 API 或工具时，结合 MCP 工具使用。
数据分析（调用数据库）、内容生成（调用搜索引擎）、自动化（调用第三方 API）

```markdown
# 数据分析 Skill

## 工作流程
1. 使用 database-query MCP 工具获取数据
2. 数据清洗：处理缺失值和异常值
3. 分析：按需求进行统计分析
4. 可视化：使用 scripts/generate-chart.py 生成图表
5. 输出报告

## MCP 工具使用规范
- 查询前先用 `EXPLAIN` 检查查询计划
- 单次查询不超过 10,000 行
- 大数据量用分页查询
```

#### 多 Agent 协作型（Swarm Pattern）

多个 Agent 各自负责一个环节，像蜂群一样协作完成复杂任务。
大型项目、需要多种专业能力的任务

```
用户需求
    │
    ▼
主编 Agent（编排者）
    ├── 调研 Agent（负责信息收集）
    ├── 写作 Agent（负责内容创作）
    ├── 审校 Agent（负责质量检查）
    └── 配图 Agent（负责视觉设计）
    │
    ▼
最终产出
```

- 每个 Agent 有独立的 Skill
- 主编 Agent 的 Skill 负责任务分配和结果整合
- Agent 之间通过文件传递数据

#### 思维蒸馏型（Distillation Pattern）

把专家的思维方式蒸馏成 Skill，让 AI 以专家的视角和决策框架来思考问题。这是 2026 年出现的最新、最有影响力的设计模式。

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1. 观察 | 观察专家如何工作、如何决策 | 原始笔记 |
| 2. 提取 | 提取专家"不假思索"的决策模式 | 决策规则列表 |
| 3. 结构化 | 把决策规则转化为条件判断 + 行动指令 | SKILL.md 草稿 |
| 4. 测试 | 用 10+ 个场景测试，与专家输出对比 | 测试报告 |
| 5. 迭代 | 根据差异修正指令 | 优化后的 SKILL.md |

- 输入：任何人的名字或主题
- 过程：搜索该人的公开资料 → 提取思维模式 → 生成 Skill
- 输出：一个以该人视角回答问题的 Skill
- 衍生：乔布斯 Skill、马斯克 Skill、芒格 Skill、费曼 Skill、纳瓦尔 Skill 等系列

它实现了知识的"民主化"——以前只有少数人能接触到顶级专家的思维方式，现在通过 Skill，任何人都能获得这种思维框架。

#### 模式组合：复杂场景的解决方案

实际项目中，往往需要组合多种设计模式。

```
1. [多方案选择型] 技术方案选型
2. [流水线型] 开发流程（设计→编码→测试→文档）
3. [检查清单型] 代码审查
4. [外部API集成型] CI/CD 部署
5. [检查清单型] 发布前检查
6. [多Agent协作型] 多团队协同发布
```

先识别任务的本质模式，再选择对应的设计模式。复杂的任务往往是多个简单模式的组合。

### 10、Skills 的未来：从工具到生态

#### 从个人工具到团队知识

Skills 的第一个进化方向是从个人效率工具变成团队知识管理工具。

| 阶段 | 形态 | 价值 | 现状 |
|------|------|------|------|
| 个人效率 | 个人 Skills 目录 | 个人工作效率提升 | 已实现 |
| 团队知识 | 团队共享 .skills/ 目录 | 团队知识不因人员流动丢失 | 正在发展 |
| 组织资产 | 公司级 Skills 仓库 | 组织级的知识沉淀和标准化 | 未来方向 |

- 新人入职：装上团队 Skills，立刻按团队标准工作
- 跨团队协作：不同团队的 Skills 互相参考
- 知识传承：老员工离职，他创建的 Skills 永远留在团队仓库里

#### Skills 经济：创作者生态

Skills 可能催生一个新的创作者经济——就像 App Store 催生了移动应用经济。

| 层次 | 内容 | 例子 |
|------|------|------|
| 第一层：免费开源 | 个人开发者分享 Skills | GitHub 上的开源 Skills |
| 第二层：付费精品 | 高质量的专业 Skills | 付费的行业分析 Skill |
| 第三层：平台级服务 | 企业级 Skills 管理平台 | Skills 的私有部署和管理 |

- 创建者：把专业知识变成 Skills，获得收入或影响力
- 使用者：安装 Skills 获得专业能力，节省学习成本
- 平台：提供 Skills 的发现、安装、更新、付费等基础设施

#### 蒸馏宇宙与"人"

蒸馏宇宙的终极形态不只是蒸馏工作流程，而是蒸馏"人"——把一个人的完整思维方式、价值观、决策框架变成 Skill。

| 深度 | 蒸馏什么 | 例子 |
|------|----------|------|
| 浅层 | 工作流程 | 审校流程、代码审查清单 |
| 中层 | 决策框架 | "面对技术选型时怎么思考" |
| 深层 | 思维方式和价值观 | "乔布斯会怎么看待这个产品" |

当你用"费曼 Skill"来理解一个物理概念时，AI 不是在模仿费曼说话，而是用费曼"从简单到复杂、用类比解释抽象"的思维框架来帮你理解。这是对人类智慧的一种数字化保存。

- 当一个人的思维方式可以被蒸馏成 Skill，"个人影响力"的定义是否改变了？
- 当你可以用任何人的思维框架来分析问题，"独立思考"意味着什么？
- 当专家知识变成可复制的数字资产，"经验"的价值是增加了还是减少了？

AI 越强大，我们越要守住底线。技术可以没有温度，但人类必须有尊严。Skills 不是替代人类，而是把人类最好的思维方式保存和传播。

#### Skills + MCP = 完整的 AI Agent 能力栈

Skills 和 MCP 的组合构成了 AI Agent 的完整能力栈——MCP 提供工具能力（手），Skills 提供工作流程（脑）。

```
用户需求
    │
    ▼
AI Agent（大脑：大语言模型）
    │
    ├── Skills 层（知道怎么做）
    │   ├── 流程指令
    │   ├── 质量标准
    │   └── 决策规则
    │
    └── MCP 层（能做什么）
        ├── 数据库连接
        ├── 文件系统操作
        ├── API 调用
        └── 外部工具
```

#### Agent 自己创建 Skills

未来的一个重要方向是 AI Agent 自己观察工作模式，自动提取并创建 Skills。

```
1. 观察：Agent 记录自己执行任务的过程
2. 分析：识别重复出现的工作模式和决策点
3. 提取：把模式转化为条件判断 + 行动指令
4. 生成：自动创建 SKILL.md
5. 测试：在新任务上测试效果
6. 优化：根据效果迭代改进
```

这意味着 Skills 的创建从"人写"→"人描述、AI 写"→"AI 自己观察并写"，门槛持续降低。
目前 Agent 自创 Skills 的质量还不够高。人类创建的 Skill 更有洞察力，因为人类理解"为什么这样做"，而 AI 目前还停留在"模仿做了什么"的层面。

#### 作者的判断

1. **Skills 会成为 AI 工具的标配**：就像每个浏览器都有扩展系统一样，每个 AI 工具都会支持 Skills
2. **蒸馏宇宙会成为新的人才市场**：你的专业能力可以被蒸馏成 Skill 并产生价值
3. **Skills + MCP 是 AI Agent 的标准架构**：未来开发 AI 产品，Skills 和 MCP 是基础设施
4. **非技术人员也能创建 Skills**：用自然语言写，用 skill-creator 辅助，创建门槛会持续降低
5. **Skills 改变知识传承方式**：团队知识不再只存在于人脑中，而是变成可安装、可更新的数字资产

## 关键句

> Skills 不是让 AI 变得更强，而是让 AI 用你的方式变强。

> AI 工具是操作系统，Skills 是应用程序——但创建 Skill 不需要写代码。

> Skills 串联用文件系统做传送带，前一个产出直接喂给下一个。

> 元数据层约 100 字永远加载，Body 层触发时加载，引用文件按需加载——这就是渐进式披露架构。

> System Prompt 是静态的全局行为规范，Skills 是动态加载的领域专家。

> MCP 提供工具能力（能做什么），Skills 提供工作流程（怎么做）——互补而非替代。

> 三遍审校不是 AI 的发明，而是人类编辑的工作习惯——Skill 的价值在于把隐性习惯变成可重复的显性流程。

> MUST/ALWAYS/NEVER 不如解释 why 有效。

> 好 Skill 不是越详细越好——精简的、解释 why 的原则式引导效果最好。

> Skill 太长会挤占任务空间——控制 SKILL.md 在 500 行以内。

> 全局级 Skills 是个人偏好，项目级 Skills 是团队规范。

> 团队的 .skills/ 目录就是团队的知识库——新人装上就能上手。

> 蒸馏宇宙的终极愿景——把人类最好的思维方式数字化保存和传播。

> Skills + MCP = 完整的 AI Agent 能力栈——MCP 是手，Skills 是脑。

> 蒸馏不是复制，而是提取思维方式。

> 创建 Skill 不是记录步骤，而是把你的思维方式结构化。

## 想法与感悟

**读前预期**：以为 Skills 只是又一个 AI 提示词技巧，和 System Prompt 差不多。

**读后收获**：Skills 的本质是"工作流的标准化封装"，它解决的不是"AI 能不能做"的问题，而是"AI 怎么按你的方式做"的问题。

**三大收获**：

1. **渐进式披露是天才设计**：三层加载架构完美解决了"想要多装 Skills"和"上下文有限"的矛盾。100 token 的 metadata 就能让 50 个 Skills 和平共处。

2. **金发姑娘区间颠覆认知**：一直以为给 AI 的指令越详细越好，ETH Zurich 的研究证明精简的原则式引导反而效果最好。这对所有 prompt engineering 都有启示。

3. **蒸馏宇宙是最深远的意义**：Skills 不只是一个效率工具，它可能是人类知识传承方式的一次革命。把专家的思维方式蒸馏成可安装的数字资产，这个概念超越了工具层面。
